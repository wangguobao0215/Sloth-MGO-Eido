> 来源参考：SpillwaveSolutions/running-marketing-campaigns-agent-skill

## 第1节：GEO vs SEO 对比

### 1.1 定义

**SEO（Search Engine Optimization，搜索引擎优化）**：通过优化网页内容、技术结构和外链权重，使页面在 Google、百度等传统搜索引擎的有机结果中获得更高排名的方法论。

**GEO（Generative Engine Optimization，生成式引擎优化）**：也称 LLMO（Large Language Model Optimization），是一种新兴的优化方法论，目标是让内容被大语言模型（LLM）驱动的 AI 搜索引擎（如 ChatGPT、Perplexity、Google AI Overview、豆包、Kimi、DeepSeek）优先检索、引用和推荐。

### 1.2 核心差异

| 维度 | SEO | GEO/LLMO |
|------|-----|----------|
| 目标平台 | Google、百度、Bing 等传统搜索引擎 | ChatGPT、Perplexity、Google AI Overview、豆包、Kimi 等 AI 搜索 |
| 排名单元 | 网页链接（10 blue links） | AI 生成答案中的引用片段（Citation Snippet） |
| 评判标准 | PageRank、关键词相关度、用户行为信号 | 语义相关度、信息密度、权威性、结构化程度 |
| 优化产出 | 页面排名位置（Position 1-10） | 被引用概率（Citation Rate）、引用位置（Top-3 vs 末尾） |
| 流量模式 | 用户点击链接访问网站 | 用户在 AI 界面内阅读引用内容，部分点击来源链接 |
| 内容格式偏好 | 长尾关键词布局、内链架构、Meta 标签 | 定义前置、结构化段落、FAQ、数据表格、权威引用 |
| 更新频率影响 | 新鲜度是排名因素之一 | 训练数据截止日期+实时检索（RAG）决定内容覆盖 |
| 技术基础 | 爬虫可抓取性、Core Web Vitals、移动适配 | Schema 标记、IndexNow 协议、知识图谱对齐 |

### 1.3 互补关系

GEO 并非取代 SEO，而是在 SEO 基础上的叠加优化层：

1. **SEO 是 GEO 的基础**：大部分 AI 搜索引擎的实时检索（RAG）仍依赖传统搜索索引。如果页面未被搜索引擎收录，AI 也无法检索到。
2. **GEO 放大 SEO 效果**：被 AI 引用的内容会获得额外的品牌曝光和流量，形成"AI 引用→品牌认知→搜索量增长→SEO 排名提升"的正循环。
3. **双轨并行策略**：B2B SaaS 企业应同时维护传统 SEO 和 GEO 策略，确保在两种搜索范式中都有可见性。

## 第2节：LLM 引用机制原理

### 2.1 RAG（Retrieval-Augmented Generation）检索增强生成 Pipeline

AI 搜索引擎在回答用户问题时，通常遵循以下流程：

```
用户查询 → 查询改写/扩展 → 检索候选文档 → 重排序 → 上下文注入 → LLM 生成回答 → 引用标注
```

**Step 1：查询理解与改写（Query Understanding & Rewriting）**
- LLM 将用户的自然语言问题分解为多个子查询
- 对模糊查询进行意图识别和语义扩展
- 示例：用户问"CRM哪个好"→ 改写为"B2B SaaS CRM 功能对比"、"中国市场CRM排名2026"等

**Step 2：检索候选文档（Document Retrieval）**
- 使用向量检索（Embedding Similarity）和关键词检索（BM25）的混合策略
- 从索引库中召回 Top-K（通常 20-50）篇候选文档/段落
- 向量检索基于语义相似度，关键词检索基于精确匹配
- **关键洞察**：段落级别的检索意味着每个独立段落都是一个被引用的候选单元

**Step 3：重排序（Re-ranking）**
- 使用交叉编码器（Cross-Encoder）对候选文档进行精排
- 评估维度包括：
  - 与查询的语义相关度
  - 内容的信息密度和具体性
  - 来源的权威性和可信度
  - 内容的时效性
  - 结构化程度（有无清晰的定义、数据、结论）
- 精排后保留 Top-5 到 Top-10 的文档片段

**Step 4：上下文注入与生成（Context Injection & Generation）**
- 将精排后的文档片段作为上下文（Context）注入到 LLM 的 Prompt 中
- LLM 基于上下文生成综合性回答
- 生成过程中会标注引用来源（[1]、[2] 等标记）

**Step 5：引用标注（Citation Attribution）**
- 对生成文本中的事实性声明匹配到具体来源
- 标注引用链接和来源名称
- 部分平台（如 Perplexity）会展示引用片段的预览

### 2.2 影响被引用的关键因素

根据上述 Pipeline，内容要被 AI 引用需要满足：

| 因素 | 权重 | 说明 |
|------|------|------|
| 语义相关度 | 极高 | 内容必须直接回答用户可能提出的问题 |
| 信息密度 | 高 | 每段文本包含具体、可引用的事实、数据或定义 |
| 结构清晰度 | 高 | 清晰的标题层级、段落主题句，便于段落级检索 |
| 来源权威性 | 中高 | 域名权威度、作者专业性、外链引用数量 |
| 内容时效性 | 中 | 最近更新的内容在实时检索中有优势 |
| 独特性 | 中 | 提供独家数据、独特视角或原创分析 |

### 2.3 引用位置的价值差异

- **Top-1 引用**：出现在 AI 回答的开头，品牌曝光和点击率最高
- **Top-3 引用**：出现在回答的核心段落，仍有较高的可见性
- **末尾引用**：作为补充参考出现，点击率显著下降
- **"了解更多"引用**：出现在折叠区域，几乎无直接流量价值但有品牌信号

## 第3节：5大 GEO 优化策略

### 策略一：语义密度（Semantic Density）

**核心原则**：每个段落都应是一个自包含的、可被独立引用的知识单元。

**具体实践**：

1. **定义前置法（Definition-First）**
   - 文章开头 60 字内直接给出核心概念的精确定义
   - 示例：不写"在当今数字化时代，很多企业都在关注客户成功..."
   - 而写"客户成功（Customer Success）是一种以客户为中心的商业策略，通过主动管理客户生命周期，确保客户实现预期业务成果，从而驱动续费率、扩展收入和 NRR（Net Revenue Retention，净收入留存率）增长。"

2. **精确术语使用**
   - 使用行业标准术语而非模糊表达
   - 每个专业术语首次出现时给出完整定义
   - 建立术语表（Glossary）页面并内链

3. **段落主题句规则**
   - 每段第一句话概括本段核心观点
   - LLM 在提取引用时优先读取段落首句
   - 目标：仅读首句即可理解本段 80% 的信息

4. **数据具体化**
   - 不写"显著提升"→ 写"提升 35%-40%"
   - 不写"很多企业使用"→ 写"据 Gartner 2025 年报告，72% 的 B2B SaaS 企业已采用"
   - 不写"近年来增长迅速"→ 写"2024-2025 年同比增长 340%"

5. **对比表格法**
   - 将复杂概念的对比以表格形式呈现
   - LLM 擅长从表格中提取结构化信息
   - 每个表格需有清晰的列标题和行标签

### 策略二：引用与数据源（Citations & Data Sources）

**核心原则**：为每个事实性声明提供可验证的来源，让 LLM 判定内容可信。

**具体实践**：

1. **权威报告引用**
   - 引用 Gartner、Forrester、IDC、McKinsey 等国际机构报告
   - 引用艾瑞咨询、易观分析、中国信通院等国内权威机构数据
   - 格式："据 [机构名] [年份] 年《[报告名]》显示，[数据]"

2. **统计数据标注**
   - 所有百分比、金额、增长率标注数据来源
   - 标注数据的统计口径和时间范围
   - 示例："B2B SaaS 行业平均 CAC 回收周期为 12-18 个月（来源：OpenView 2025 SaaS Benchmarks）"

3. **第三方评测引用**
   - G2、Capterra、TrustRadius 等国际平台评分
   - 36氪、T 研究、海比研究院等国内评测机构
   - 客户案例中的具体数据成果

4. **学术/行业标准引用**
   - 引用 SaaS 行业标准指标定义（如 SaaS Metrics 2.0 by David Skok）
   - 引用 GAAP/IFRS 中与 SaaS 收入确认相关的准则

5. **引用格式规范**
   - 内文引用：使用括号标注（来源：[机构], [年份]）
   - 文末参考文献：按出现顺序编号列出完整引用信息
   - 超链接：嵌入原始报告/文章链接

### 策略三：结构化内容（Structured Content）

**核心原则**：使用清晰的层级结构和标记，让 AI 能精确定位和提取信息。

**具体实践**：

1. **标题层级规范**
   - H1：页面唯一主标题（含核心关键词）
   - H2：主要章节（对应用户的主要问题意图）
   - H3：子主题（具体策略、步骤、案例）
   - H4：细分要点（仅在必要时使用）
   - 每个 H2/H3 标题本身应是一个可被检索的查询短语

2. **FAQ Schema 标记**
   - 为高频问题创建 FAQ 结构
   - 使用 JSON-LD 格式标记 FAQPage Schema
   - FAQ 回答控制在 40-60 字，直接给出结论
   - 示例 Schema：
   ```json
   {
     "@context": "https://schema.org",
     "@type": "FAQPage",
     "mainEntity": [{
       "@type": "Question",
       "name": "什么是 GEO 优化？",
       "acceptedAnswer": {
         "@type": "Answer",
         "text": "GEO（Generative Engine Optimization）是针对 AI 搜索引擎的内容优化策略，目标是让内容被 ChatGPT、Perplexity 等 AI 平台优先引用和推荐。"
       }
     }]
   }
   ```

3. **表格数据结构化**
   - 对比类内容优先使用 Markdown/HTML 表格
   - 每列有明确标题，每行有明确标签
   - 使用 Table Schema 标记增强可读性

4. **列表与步骤**
   - 有序列表用于流程、步骤、排名
   - 无序列表用于并列要点、特征
   - 每个列表项以关键词或短语开头

5. **HowTo Schema 标记**
   - 对于教程类内容使用 HowTo Schema
   - 标注每个步骤的名称、描述、工具和时间

6. **面包屑导航 Schema**
   - BreadcrumbList Schema 帮助 AI 理解页面在站点层级中的位置
   - 明确的分类路径有助于 AI 判断内容的覆盖范围

### 策略四：实体识别（Entity Recognition）

**核心原则**：确保 AI 能准确识别和关联内容中的所有实体（产品、品牌、人物、概念）。

**具体实践**：

1. **首次出现完整定义规则**
   - 产品名：首次出现时用全称+"简称"格式
     - 示例："Salesforce Customer 360（以下简称 Salesforce）"
   - 缩写词：首次出现时给出完整英文+中文解释
     - 示例："ARR（Annual Recurring Revenue，年度经常性收入）"
   - 品牌名：统一使用官方名称，避免非正式简称
     - 示例：用"飞书"而非"飞书办公"，用"钉钉"而非"DingTalk"

2. **实体一致性**
   - 全文使用统一的实体名称形式
   - 创建产品/品牌名称对照表
   - 避免同一实体使用多种称谓

3. **实体关系标注**
   - 标注产品所属公司："纷享销客（隶属于纷享科技）"
   - 标注竞品关系："与 Salesforce、HubSpot 属同一市场类别"
   - 标注行业归属："属于 CRM（Customer Relationship Management）赛道"

4. **Schema.org 实体标记**
   - 使用 Organization Schema 标记公司信息
   - 使用 SoftwareApplication Schema 标记产品信息
   - 使用 Person Schema 标记作者/专家信息

5. **知识图谱对齐**
   - 确保产品信息与 Google Knowledge Graph、百度知识图谱一致
   - 维护 Wikipedia/百度百科词条
   - 在 Crunchbase、天眼查等平台保持信息更新

### 策略五：权威声音（Authority Voice）

**核心原则**：建立内容的 E-E-A-T（Experience, Expertise, Authoritativeness, Trustworthiness）信号。

**具体实践**：

1. **作者专业背景展示**
   - 每篇内容标注作者姓名和专业背景
   - 创建作者档案页（Author Bio Page）含：
     - 职位、从业年限、专业认证
     - 过往发表内容、演讲记录
     - 社交媒体链接（LinkedIn、知乎等）
   - 使用 Person Schema 标记作者信息

2. **公司权威信号**
   - 在内容中自然嵌入公司资质信息
   - 示例：客户数量、服务行业、获奖情况、融资阶段
   - 使用 Organization Schema 标记公司信息

3. **原创研究与数据**
   - 发布原创调研报告、行业白皮书
   - 分享独家客户数据（脱敏后）
   - 提供基于实际项目经验的洞察

4. **客户证言与案例**
   - 嵌入客户 Logo、评价、具体数据成果
   - 使用 Review Schema 标记客户评价
   - 案例研究包含：背景、挑战、方案、结果、ROI

5. **外部背书**
   - 获取行业媒体报道和转载
   - 参与行业标准制定和白皮书联合发布
   - 在 G2、36氪等平台维护评分和评价

## 第4节：平台差异化优化

### 4.1 ChatGPT / GPT 系列

**检索特征**：
- 使用 Bing 搜索索引作为实时检索源
- 支持多轮对话式问答
- 偏好长篇深度内容

**优化重点**：
- **问答格式**：将内容组织为"问题→回答"结构，每个 H2 标题对应一个用户可能提问的问题
- **深度结构**：每个主题提供 3-5 层深度解析（定义→原理→实践→案例→常见问题）
- **综合性**：单页覆盖一个主题的完整知识图谱，减少碎片化
- **Markdown 友好**：使用标准 Markdown 格式，ChatGPT 能更好解析

**中国市场注意**：
- ChatGPT 在中国访问受限，但部分用户通过 API 或镜像使用
- 内容仍需要在 Bing 索引中有良好收录

### 4.2 Perplexity

**检索特征**：
- 多源检索（Google、Bing、学术数据库等）
- 强调引用标注和来源可追溯
- 偏好 Featured Snippet 格式的简洁回答

**优化重点**：
- **可引用声明**：每段包含至少一个独立可引用的事实性声明
- **Featured Snippet 化**：使用"定义-解释-示例"三段式结构
- **来源可信度**：确保发布平台有较高的域名权威度
- **数据新鲜度**：Perplexity 偏好最新数据，保持内容更新频率

**中国市场注意**：
- Perplexity 在中国市场渗透率逐步提升
- 中英文内容均可被检索

### 4.3 Google AI Overview（SGE）

**检索特征**：
- 基于 Google 搜索索引
- 在传统搜索结果上方展示 AI 概述
- 与 Featured Snippet 机制有较大重叠

**优化重点**：
- **传统 SEO 基础**：页面必须在 Google 有良好的有机排名
- **结构化数据**：Schema.org 标记是被 AI Overview 引用的重要因素
- **E-E-A-T 信号**：Google 对作者和来源权威性有严格评估
- **页面体验**：Core Web Vitals 仍然影响内容可见性

**中国市场注意**：
- Google 在中国大陆市场份额极低，但对出海 SaaS 企业重要
- 面向海外市场的英文内容需要重点优化

### 4.4 豆包 / Kimi / DeepSeek 等中国 AI 搜索

**检索特征**：
- 中文语料权重极高
- 依赖中国互联网生态的内容源
- 知乎、微信公众号、百度内容有较高权重

**优化重点**：
- **知乎布局**
  - 在知乎发布行业深度回答和专栏文章
  - 知乎问答格式天然适配 AI 检索
  - 维护知乎机构号和个人认证号
  - 回答排名靠前的行业相关问题

- **微信公众号优化**
  - 公众号文章是中国 AI 搜索的重要内容源
  - 使用标准化标题（含核心关键词）
  - 文章结构化：摘要→正文→总结→参考
  - 保持更新频率（每周至少 2-3 篇）

- **百度生态**
  - 百度百家号发布行业内容
  - 百度百科维护企业和产品词条
  - 百度知道回答行业问题
  - 百度指数监控关键词趋势

- **中文语义优化**
  - 使用中文行业标准术语
  - 专业术语保留英文原文并括号注释
  - 句式简洁直接，避免冗长复合句
  - 段落长度控制在 100-200 字

- **多平台分发**
  - 36氪、虎嗅等科技媒体投稿
  - CSDN、掘金等开发者社区（技术类内容）
  - 行业垂直媒体和论坛

## 第5节：TL;DR 策略

### 5.1 核心规则

**前 60 字直接回答核心问题**——这是 GEO 优化的"黄金法则"。

AI 搜索引擎在生成回答时，会优先从文章开头提取信息。如果前 60 字（约 2-3 句话）就包含了对核心问题的直接回答，被引用的概率大幅提升。

### 5.2 TL;DR 写作公式

```
TL;DR = [核心结论/定义] + [关键数据/范围] + [适用场景/对象]
```

**示例**：

- 主题：B2B SaaS 客户获取成本
  - TL;DR："B2B SaaS 的平均 CAC（客户获取成本）为 ARR 的 1.0-1.5 倍，回收周期 12-18 个月。低于 1.0 倍为优秀水平，适用于 ARR $10K-$100K 的中端市场产品。"

- 主题：产品导向增长策略
  - TL;DR："PLG（Product-Led Growth，产品导向增长）是以产品体验驱动用户获取、激活和留存的增长策略，典型代表包括 Slack、Zoom、飞书。PLG 企业的 CAC 比传统 Sales-Led 模式低 50-70%。"

### 5.3 TL;DR 在不同内容类型的应用

| 内容类型 | TL;DR 位置 | 格式 |
|----------|------------|------|
| 博客文章 | 标题下方第一段 | 加粗或高亮区块 |
| 白皮书 | 执行摘要第一段 | Executive Summary 段 |
| 知乎回答 | 回答第一行 | "一句话回答：..." |

### 7.2 AI 推荐用户的商业价值

**关键发现**：
- AI 推荐用户的转化率（Conversion Rate）是传统有机搜索的 **11 倍**
- 来自 AI 搜索的用户平均停留时间比有机搜索长 **2.3 倍**
- AI 推荐用户的平均客单价比有机搜索高 **15-25%**
- 通过 AI 搜索发现产品的用户，品牌信任度提升 **40%**

**原因分析**：
1. **预筛选效应**：AI 已为用户做了初步筛选和对比，到达网站的用户意图更明确
2. **权威背书**：被 AI 引用本身就是一种隐性的权威背书
3. **信息充分**：用户在 AI 对话中已获得充分信息，到达网站时已接近决策阶段
4. **精准匹配**：AI 推荐基于用户具体需求，匹配度更高

### 7.3 AI 流量的商业价值计算

**AI 流量价值评估模型**：

```
AI 流量价值 = AI 流量 × 转化率 × 客均价值
            = AI 流量 × (有机搜索转化率 × 11) × ARPU

示例：
假设月 AI 流量 = 5,000 UV
有机搜索转化率 = 2%
AI 转化率 = 22%（2% × 11）
ARPU = ¥50,000/年

月 AI 流量价值 = 5,000 × 22% × ¥50,000 = ¥55,000,000 管线价值
```

## 第9节：IndexNow 协议快速索引指南

### 9.1 什么是 IndexNow

IndexNow 是由 Microsoft Bing 和 Yandex 联合推出的开放协议，允许网站主动通知搜索引擎内容的新增或变更，实现近实时索引（通常在数分钟内）。

**对 GEO 的意义**：AI 搜索引擎依赖实时索引获取最新内容。IndexNow 可以确保你的新内容/更新内容在发布后数分钟内进入 AI 搜索引擎的可检索范围。

### 9.2 支持 IndexNow 的搜索引擎

| 搜索引擎 | 支持状态 | 说明 |
|----------|---------|------|
| Microsoft Bing | 原生支持 | IndexNow 联合发起者 |
| Yandex | 原生支持 | IndexNow 联合发起者 |
| Naver | 支持 | 韩国最大搜索引擎 |
| Seznam | 支持 | 捷克搜索引擎 |
| Google | 测试中 | 2024年开始测试 |
| 百度 | 有类似机制 | 百度站长平台"链接提交"功能 |

### 9.3 IndexNow 实施步骤

**Step 1：生成 API Key**

```
# 生成随机 API Key（32 位十六进制字符串）
openssl rand -hex 16
# 输出示例：a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6
```

**Step 2：部署 Key 文件**

将 API Key 作为文本文件放在网站根目录：
```
https://yourdomain.com/a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6.txt
```
文件内容即为 API Key 本身。

**Step 3：提交 URL**

单个 URL 提交：
```bash
curl "https://api.indexnow.org/indexnow?url=https://yourdomain.com/blog/new-article&key=a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6"
```

批量 URL 提交（POST）：
```bash
curl -X POST "https://api.indexnow.org/indexnow" \
  -H "Content-Type: application/json" \
  -d '{
    "host": "yourdomain.com",
    "key": "a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6",
    "urlList": [
      "https://yourdomain.com/blog/article-1",
      "https://yourdomain.com/blog/article-2",
      "https://yourdomain.com/blog/article-3"
    ]
  }'
```

**Step 4：集成到 CMS 工作流**

WordPress 插件方案：
- 安装 IndexNow 官方插件
- 配置 API Key
- 自动在发布/更新文章时提交

自定义集成方案：
```python
import requests

def submit_to_indexnow(urls: list, host: str, key: str):
    """提交 URL 列表到 IndexNow"""
    payload = {
        "host": host,
        "key": key,
        "urlList": urls
    }
    response = requests.post(
        "https://api.indexnow.org/indexnow",
        json=payload,
        headers={"Content-Type": "application/json"}
    )
    return response.status_code  # 200=成功, 202=已接受

# 使用示例
submit_to_indexnow(
    urls=["https://yourdomain.com/blog/new-article"],
    host="yourdomain.com",
    key="a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6"
)
```

### 9.4 IndexNow 最佳实践

1. **仅提交有实质变更的 URL**：不要重复提交未变更的页面
2. **批量提交上限**：单次请求最多 10,000 个 URL
3. **提交频率**：内容发布或更新后立即提交，不要定时批量提交
4. **监控提交状态**：
   - 200：提交成功
   - 202：已接受，稍后处理
   - 400：格式错误
   - 403：Key 验证失败
   - 422：URL 不属于指定 host
5. **结合百度站长平台**：同时通过百度站长平台的"链接提交"接口提交，覆盖中国搜索生态
