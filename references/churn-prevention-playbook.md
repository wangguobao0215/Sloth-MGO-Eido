## 第1节：双重流失模型

### 1.1 流失分类

客户流失（Churn）并非单一现象，而是由两种截然不同的机制驱动：

**主动流失（Voluntary Churn / Active Churn）**
- 定义：客户有意识地决定取消订阅或停止使用产品
- 占比：总流失的 **50-70%**
- 驱动因素：产品不满、价格敏感、竞品替代、需求变化
- 可干预性：高——通过优化取消流程、挽留方案和客户成功主动干预可有效减少

**被动流失（Involuntary Churn / Passive Churn）**
- 定义：客户并非有意取消，而是因为支付失败（信用卡过期、余额不足等）导致订阅中断
- 占比：总流失的 **30-50%**
- 驱动因素：信用卡过期、余额不足、银行拒绝交易、支付方式失效
- 可干预性：高——通过 Dunning 催收栈和智能支付重试可回收 50-60% 的被动流失

### 1.2 双重流失的战略含义

```
总月流失率 = 主动流失率 + 被动流失率

B2B SaaS 健康基准：
- 总月流失率 < 2%
- 主动流失率 < 1.2%
- 被动流失率 < 0.8%

示例：
月初客户数 = 1,000
主动流失 = 12 (1.2%)
被动流失 = 6 (0.6%)
总流失 = 18 (1.8%) ✅ 低于 2% 基准

年化流失率 ≈ 1 - (1 - 1.8%)^12 ≈ 19.6%
年化留存率 ≈ 80.4%
```

### 1.3 为什么必须分开治理

| 维度 | 主动流失 | 被动流失 |
|------|---------|---------|
| 根因 | 产品/服务/价格问题 | 支付技术问题 |
| 治理团队 | 客户成功+产品 | 增长工程+财务 |
| 干预时机 | 取消前（行为预警）→取消时（挽留流程）→取消后（赢回） | 支付失败前（预催收）→失败时（重试+催收）→失败后（宽限期） |
| ROI | 中高（需产品/服务改进） | 极高（纯技术/流程优化，成本低） |
| 优先级 | 长期战略性 | 短期战术性（快速见效） |

**中国市场特殊性**：
- 中国 B2B SaaS 的被动流失比例可能更高，因为：
  - 企业客户常用对公转账，到期续费流程长
  - 微信支付/支付宝在小额 SaaS 中常见，但自动扣款限制较多
  - 年付模式下，企业预算审批周期可能导致"技术性断约"
- 建议：将"续费流程管理"作为被动流失治理的核心

## 第2节：客户健康度评分模型

### 2.1 综合健康度评分公式

```
Health Score = 登录频率(0.30) + 功能使用(0.25) + 客服情绪(0.15) + 账单健康(0.15) + 互动参与(0.15)
```

### 2.2 各维度评分规则

**维度1：登录频率（Login Frequency）—— 权重 0.30**

| 登录频率 | 分值 | 说明 |
|----------|------|------|
| 每日活跃（DAU） | 100 | 理想状态 |
| 每周 3-4 次 | 80 | 正常使用 |
| 每周 1-2 次 | 60 | 轻度使用，需关注 |
| 每月 2-4 次 | 40 | 使用下降，风险信号 |
| 每月 <2 次 | 20 | 高流失风险 |
| 30 天未登录 | 0 | 危急状态 |

**维度2：功能使用（Feature Adoption）—— 权重 0.25**

| 功能使用深度 | 分值 | 说明 |
|-------------|------|------|
| 使用 80%+ 核心功能 | 100 | 深度用户，粘性极强 |
| 使用 60-80% 核心功能 | 80 | 良好采纳 |
| 使用 40-60% 核心功能 | 60 | 中等采纳，有提升空间 |
| 使用 20-40% 核心功能 | 40 | 采纳不足，需引导 |
| 仅使用 <20% 核心功能 | 20 | 严重不足 |
| 核心功能完全未使用 | 0 | 价值未实现 |

核心功能定义方法：
- 识别产品中与客户核心价值直接相关的 5-8 个功能
- 基于数据分析确认"使用这些功能的客户续费率显著更高"
- 典型核心功能：主流程操作、数据导入/导出、报表查看、协作功能

**维度3：客服情绪（Support Sentiment）—— 权重 0.15**

| 客服互动情况 | 分值 | 说明 |
|-------------|------|------|
| 无工单/正面反馈 | 100 | 满意度高 |
| 偶尔咨询，快速解决 | 80 | 正常 |
| 频繁咨询，均已解决 | 60 | 有学习曲线问题 |
| 有投诉工单，已解决 | 40 | 存在不满 |
| 多次投诉/升级 | 20 | 高度不满 |
| 要求退款/威胁取消 | 0 | 危急 |

**维度4：账单健康（Billing Health）—— 权重 0.15**

| 账单状态 | 分值 | 说明 |
|----------|------|------|
| 按时付款，无逾期 | 100 | 健康 |
| 偶尔晚付 1-3 天 | 80 | 基本正常 |
| 有过 1 次支付失败但已恢复 | 60 | 需关注 |
| 多次支付失败/经常逾期 | 40 | 风险 |
| 当前有逾期未付 | 20 | 高风险 |
| 降级到更低计划 | 10 | 预算收缩信号 |

**维度5：互动参与（Engagement）—— 权重 0.15**

| 互动行为 | 分值 | 说明 |
|----------|------|------|
| 参加 Webinar/活动+回复邮件+NPS 反馈 | 100 | 高参与 |
| 回复邮件+偶尔参加活动 | 80 | 良好参与 |
| 仅打开邮件 | 60 | 被动参与 |
| 不打开邮件但仍在使用产品 | 40 | 低参与 |
| 完全不互动 | 20 | 脱离 |
| 退订邮件/拉黑 | 0 | 完全脱离 |

### 2.3 四级健康分级

| 分级 | 分数范围 | 状态标签 | 行动策略 |
|------|---------|---------|---------|
| 健康/可追加 | 80-100 | 绿色 | 推进扩展销售（Upsell/Cross-sell）、邀请成为案例/推荐人 |
| 关注 | 60-79 | 黄色 | 安排季度业务回顾（QBR）、针对性功能引导、发送价值报告 |
| 风险 | 40-59 | 橙色 | 客户成功经理主动外联、制定挽留计划、高管关怀 |
| 危急 | 0-39 | 红色 | 立即启动危机干预、高管 1 对 1 沟通、启动挽留方案 |

### 2.4 健康度评分实施建议

1. **数据采集自动化**：通过产品埋点、CRM 系统、客服工单系统自动采集各维度数据
2. **评分频率**：每日更新评分，每周生成健康度变化报告
3. **预警机制**：任何客户健康度连续 2 周下降 10 分以上自动触发预警
4. **校准周期**：每季度根据实际流失数据回归分析，调整各维度权重
5. **中国市场补充维度**：可增加"企微/钉钉群活跃度"作为互动参与的补充信号

## 第3节：流失风险信号矩阵

### 3.1 高风险信号（红色预警）

| 信号 | 风险级别 | 预计流失时间 | 检测方法 | 建议干预时间 |
|------|---------|------------|---------|-------------|
| 登录频率下降 >50% | 高 | 2-4 周 | 产品分析工具对比最近 14 天 vs 前 14 天 | 发现后 48 小时内 |
| 核心功能停用 | 高 | 1-3 周 | 核心功能最近 7 天使用量=0 | 发现后 24 小时内 |
| 访问账单/订阅管理页面 | 高 | 数天 | 页面访问埋点 | 发现后 24 小时内 |
| 数据批量导出 | 危急 | 数天 | API 调用日志/导出功能使用记录 | 发现后立即 |
| NPS 评分 0-6（贬损者） | 高 | 2-8 周 | NPS 调研回收 | 24 小时内个人跟进 |
| 客服投诉升级 | 高 | 1-4 周 | 客服工单升级记录 | 发现后 24 小时内 |
| 合同即将到期（<30 天）未续签 | 高 | <30 天 | CRM 合同到期日 | 到期前 60 天开始 |

### 3.2 中风险信号（黄色关注）

| 信号 | 风险级别 | 预计流失时间 | 检测方法 | 建议干预时间 |
|------|---------|------------|---------|-------------|
| 登录频率下降 20-50% | 中 | 4-8 周 | 产品分析 | 发现后 1 周内 |
| 活跃用户数减少 | 中 | 4-12 周 | 团队账号活跃度 | 发现后 2 周内 |
| 未参加季度业务回顾 | 中 | 不确定 | QBR 日历 | 安排补充沟通 |
| 关键联系人离职 | 中 | 4-12 周 | CRM 联系人状态/LinkedIn 监控 | 发现后 1 周内 |
| 支持工单增加但满意度下降 | 中 | 4-8 周 | 客服系统 | 发现后 1 周内 |
| 新功能未采纳 | 中 | 长期 | 功能使用追踪 | 下次互动时引导 |

### 3.3 早期信号（蓝色监控）

| 信号 | 风险级别 | 说明 | 建议行动 |
|------|---------|------|---------|
| 邮件打开率持续下降 | 低 | 参与度降低的早期信号 | 优化邮件内容/频率 |
| 社区/论坛不再活跃 | 低 | 社群参与度降低 | 邀请参加线上活动 |
| 竞品安装/试用（可检测时） | 低-中 | 评估替代方案 | 主动沟通差异化价值 |
| 行业/公司负面新闻 | 低 | 可能影响预算 | 准备灵活方案 |
| 产品使用模式异常 | 低 | 可能在测试竞品或准备迁移 | 深入分析行为数据 |

### 3.4 复合信号预警规则

当多个信号同时出现时，风险叠加：

```
Rule 1: 登录下降 >50% AND 访问账单页 → 危急（72 小时内干预）
Rule 2: 核心功能停用 AND 数据导出 → 危急（24 小时内干预）
Rule 3: NPS 0-6 AND 客服投诉 → 高风险（48 小时内干预）
Rule 4: 关键联系人离职 AND 登录下降 → 高风险（1 周内干预）
Rule 5: 登录下降 AND 邮件不再打开 AND 未参加 QBR → 高风险（1 周内干预）
```

## 第4节：取消流程设计

### 4.1 取消流程全链路

```
触发取消意图 → 离开调查 → 动态挽留方案 → 确认决定 → 执行取消 → 取消后跟进
     ↑              ↓              ↓              ↓           ↓            ↓
  产品内按钮    收集原因       个性化方案      最后确认    保留数据      赢回序列
  或联系客服    必填单选       基于原因匹配    冷却期选项  导出选项      30/60/90天
```

### 4.2 取消触发点设计

- 取消按钮位置：设置页 → 订阅管理 → 取消订阅
- **不要隐藏取消按钮**：这会导致客户投诉和负面口碑，且在中国市场可能违反消费者权益保护规定
- 点击"取消"后进入取消流程（而非直接取消）

### 4.3 离开调查（Exit Survey）

**7 种标准离开原因**：

| 编号 | 原因 | 对应标签 | 频率基准 |
|------|------|---------|---------|
| 1 | 太贵了 / 预算不足 | price | 25-35% |
| 2 | 使用不够多 / 没时间用 | underuse | 15-25% |
| 3 | 缺少我需要的功能 | missing_feature | 15-20% |
| 4 | 切换到其他产品 | competitor | 10-15% |
| 5 | 遇到技术问题 / 不稳定 | technical | 5-10% |
| 6 | 只是季节性/临时需要 | seasonal | 5-10% |
| 7 | 业务关闭/不再需要 | business_closed | 5-10% |

**调查设计原则**：
- 必填单选（不能跳过）
- 可选补充文本框（"请告诉我们更多"）
- 语气友好不施压
- 不要超过 2 步

### 4.4 动态挽留方案（基于离开原因）

每种离开原因触发对应的个性化挽留页面：

**原因1：太贵了**
```
展示内容：
- "我们理解预算限制。以下方案或许对你有帮助："
- 方案A：享受 [20-30%] 折扣，为期 [2-3] 个月
- 方案B：降级到 [更低价格的计划名]，保留核心功能
- 方案C：切换到年付方案，节省 [X%]
- 展示使用数据："过去30天，你的团队使用我们节省了约 [X] 小时"
```

**原因2：使用不够多**
```
展示内容：
- "我们可以帮你获得更多价值："
- 方案A：安排 1对1 使用指导（15分钟，免费）
- 方案B：暂停账户 1-3 个月（保留数据，随时重启）
- 方案C：发送"3分钟上手指南"
- 展示提示："你可能还没试过 [未使用的核心功能名]"
```

**原因3：缺少功能**
```
展示内容：
- "你需要的功能可能正在路线图中："
- 输入框：请描述你需要的功能
- 方案A：路线图预览 + 功能上线后优先通知
- 方案B：申请 Beta 测试资格
- 方案C：暂停等待功能上线
- 展示："以下功能即将在 [季度] 上线：[功能列表]"
```

**原因4：切换到竞品**
```
展示内容：
- "在你离开前，我们想确保你做了全面对比："
- 展示竞品对比表（差异化要点）
- 方案A：安排与产品专家的迁移咨询（帮助评估）
- 方案B：个性化外联——客户成功经理直接联系
- 不要贬低竞品，保持专业态度
```

**原因5：技术问题**
```
展示内容：
- "对不起给你带来不便，让我们来解决："
- 方案A：立即安排专属技术支持（优先级提升）
- 方案B：技术团队 24 小时内响应
- 方案C：如问题无法解决，承诺退款/补偿
- 展示："你报告的问题 [问题描述] 的当前处理状态"
```

**原因6：季节性需求**
```
展示内容：
- "我们完全理解！你可以选择暂停而非取消："
- 方案A：暂停账户（保留所有数据和设置，免费）
- 方案B：降级到免费/最低层计划
- 暂停期限：1-6 个月可选
- 展示："暂停期间你的数据将完整保留，随时可一键重启"
```

**原因7：业务关闭**
```
展示内容：
- "祝你未来一切顺利。在你离开前："
- 方案A：一键导出所有数据（提供多种格式）
- 方案B：如果只是转型，我们有 [其他产品/计划] 可能适合
- 保持温暖友好的告别语气
- 收集推荐意愿："如果未来有同行需要，你愿意推荐我们吗？"
```

### 4.5 确认与冷却期

- 选择挽留方案后回到产品
- 坚持取消则进入最终确认页
- 最终确认页展示：
  - 取消后会失去什么（具体功能和数据）
  - 取消生效日期（当前计费周期结束）
  - 数据保留期限（通常 30-90 天）
  - "确认取消"按钮

### 4.6 取消后跟进序列

| 时间点 | 动作 | 内容 |
|--------|------|------|
| 取消当天 | 确认邮件 | 确认取消生效日期+数据导出链接+重新激活链接 |
| 取消后 7 天 | 反馈请求 | "我们能做什么来改进？"（简短调查） |
| 取消后 30 天 | 赢回邮件1 | 产品更新亮点+特别回归优惠 |
| 取消后 60 天 | 赢回邮件2 | 客户成功案例+行业趋势 |
| 取消后 90 天 | 最后赢回 | 限时特别优惠+数据即将过期提醒 |
| 取消后 180 天 | 品牌保持 | 年度行业报告/白皮书分享（无销售语气） |

## 第5节：挽留方案映射表

### 5.1 完整挽留方案矩阵

| 离开原因 | 首选方案 | 备选方案 | 预期接受率 | 预期挽留成功率 |
|----------|---------|---------|-----------|-------------|
| 太贵了 | 20-30%折扣2-3个月 | 降级计划/年付优惠 | 20-30% | 15-25% |
| 使用不足 | 个性化使用指导 | 暂停1-3月 | 15-25% | 挽留10-15%/暂停重激活60-80% |
| 缺功能 | 路线图预览+优先通知 | Beta邀请/暂停等待 | 10-20% | 8-15% |
| 切换竞品 | 差异化对比+迁移咨询 | 个人外联/特别优惠 | 8-15% | 5-10% |
| 技术问题 | 专属技术支持 | 上门服务/补偿 | 25-35% | 20-30% |
| 季节性 | 暂停（保留数据） | 降级到最低计划 | 40-60% | 重激活60-80% |
| 业务关闭 | 数据导出帮助 | 推荐计划 | N/A | N/A（收集推荐） |

### 5.2 挽留方案设计原则

1. **匹配原因**：方案必须针对客户说的离开原因，不要一刀切
2. **尊重客户**：提供价值而非施压，不要让客户感到被"绑架"
3. **限时性**：折扣和特别方案应有明确期限（2-3 个月），之后恢复正常价格
4. **一次性**：同一客户的折扣挽留不应超过 2 次（否则训练客户"威胁取消=打折"）
5. **数据驱动**：持续追踪各方案的接受率和长期留存效果，优化方案配置

### 5.3 暂停机制（Pause）深度设计

暂停是 B2B SaaS 中被严重低估的挽留工具：

| 暂停参数 | 推荐配置 | 说明 |
|----------|---------|------|
| 暂停期限选项 | 1/2/3/6 个月 | 让客户自选 |
| 暂停期间费用 | 免费或名义费用（如原价10%） | 降低门槛 |
| 数据保留 | 完整保留所有数据和配置 | 核心卖点 |
| 重激活方式 | 一键重启，无需重新配置 | 降低回归摩擦 |
| 暂停到期处理 | 到期前 7 天邮件提醒 → 自动恢复或延长暂停 | 避免意外扣费 |
| 暂停期间沟通 | 每月 1 封产品更新简报 | 保持连接 |
| 预期重激活率 | **60-80%** | 远高于取消后赢回率（5-15%） |

## 第6节：Dunning 催收栈

### 6.1 什么是 Dunning

Dunning 是针对被动流失的自动化催收流程，名称来源于"dun"（催债）。其目标是在支付失败后通过技术重试和用户沟通回收订阅收入。

### 6.2 Dunning 全流程

```
预催收（Pre-dunning）
    ↓
支付失败
    ↓
智能重试（Smart Retry）
    ↓
催收邮件序列（Dunning Emails）
    ↓
宽限期（Grace Period）
    ↓
硬取消（Hard Cancel）
```

### 6.3 预催收阶段（Pre-dunning）

在支付失败发生前主动预防：

| 时间点 | 动作 | 渠道 | 内容 |
|--------|------|------|------|
| 信用卡到期前 30 天 | 更新提醒1 | 邮件+应用内通知 | "你的支付方式将于 [日期] 到期，请更新以确保服务不中断" |
| 信用卡到期前 14 天 | 更新提醒2 | 邮件+应用内横幅 | 更紧迫的语气+一键更新链接 |
| 信用卡到期前 3 天 | 最后提醒 | 邮件+短信/企微消息 | "你的支付方式即将到期，请立即更新" |
| 年付到期前 45 天 | 续费提醒 | 邮件+客户成功经理 | 续费价值回顾+续费流程说明 |
| 年付到期前 14 天 | 紧急提醒 | 邮件+电话 | 确认续费意向+解决障碍 |

### 6.4 智能重试策略（Smart Retry）

**软拒绝（Soft Decline）vs 硬拒绝（Hard Decline）**

| 拒绝类型 | 原因 | 重试策略 | 成功率 |
|----------|------|---------|--------|
| 软拒绝 | 余额不足、日限额、临时冻结 | 多次重试，间隔递增 | 40-60% |
| 硬拒绝 | 卡号无效、卡被注销、风控拦截 | 仅重试 1 次确认，然后走催收 | 5-10% |

**软拒绝重试时间表**：

```
失败后 6 小时 → 第1次重试
失败后 24 小时 → 第2次重试
失败后 72 小时 → 第3次重试
失败后 7 天 → 第4次重试（最后一次自动重试）
```

**重试时机优化**：
- 优先在上午 10:00-12:00 重试（余额充足概率更高）
- 月初（1-5 号）重试成功率高于月末（发薪周期影响）
- 避免周末和节假日重试
- 中国市场：避开双十一、618 等大促后的信用卡账单高峰期

### 6.5 催收邮件序列（Dunning Email Sequence）

**共 4 封邮件，覆盖 10 天**：

**邮件1：友好通知（失败后第 1 天）**
```
主题：⚠️ 你的 [产品名] 支付需要更新
语气：友好、信息性
内容：
- 支付未成功的事实说明
- 可能的原因（信用卡到期、余额不足）
- 醒目的"更新支付方式"按钮
- 不要使用"催收"、"欠费"等负面词汇
```

**邮件2：紧迫提醒（失败后第 3 天）**
```
主题：你的 [产品名] 账户即将受影响
语气：关切、紧迫
内容：
- 强调服务可能中断
- 列出将会失去的功能/数据
- "更新支付方式"按钮
- 提供替代支付方式选项
```

**邮件3：价值提醒（失败后第 7 天）**
```
主题：不要失去你在 [产品名] 中的 [具体数据/工作成果]
语气：价值导向
内容：
- 展示客户的使用数据（如"你的团队已创建了 X 个项目"）
- 强调数据将被保留的期限
- "更新支付方式"按钮
- 提供联系客服的选项
```

**邮件4：最后通知（失败后第 10 天）**
```
主题：最后通知：你的 [产品名] 账户将于 [日期] 关闭
语气：正式、最终
内容：
- 明确的关闭日期
- 关闭后的影响（数据保留期）
- 数据导出链接（以防万一）
- 最后的"更新支付方式"按钮
- "如需帮助请联系我们"
```

### 6.6 宽限期（Grace Period）

| 参数 | 推荐配置 | 说明 |
|------|---------|------|
| 宽限期时长 | 7-14 天 | 从第 4 封催收邮件发出后开始 |
| 宽限期内功能 | 降级为只读 | 可查看数据但不能创建/编辑 |
| 宽限期通知 | 应用内横幅+每次登录弹窗 | "你的账户处于宽限期" |
| 宽限期结束 | 关闭邮件+数据导出链接 | 最后一次沟通机会 |

### 6.7 硬取消后数据保留

| 时间段 | 数据状态 | 重激活方式 |
|--------|---------|-----------|
| 0-30 天 | 完整保留，一键恢复 | 更新支付方式即可 |
| 30-90 天 | 完整保留，需联系客服恢复 | 联系客服+更新支付 |
| 90+ 天 | 数据标记为待删除 | 联系客服尝试恢复 |
| 180+ 天 | 数据永久删除 | 重新注册（新账户） |

### 6.8 中国市场 Dunning 特殊考量

| 场景 | 挑战 | 解决方案 |
|------|------|---------|
| 对公转账 | 无法自动重试 | 提前 45 天发送续费发票+专人跟进 |
| 微信支付/支付宝 | 自动扣款限制 | 到期前推送小程序/H5 支付页面 |
| 企业预算审批 | 审批周期长 | 提供 60 天延长宽限期选项 |
| 财务年度切换 | 跨年续费延迟 | Q4 提前沟通次年预算 |
| 多种支付方式 | 切换支付方式复杂 | 提供支付宝/微信/银行转账等多选项 |

## 第7节：主动干预策略

### 7.1 基于行为触发的自动干预

| 触发条件 | 干预动作 | 负责人 | 响应时间 |
|----------|---------|--------|---------|
| 使用量下降 >50%（连续 2 周） | 发送"再参与"邮件+使用指南 | 营销自动化 | 自动触发 |
| 核心功能 7 天未使用 | 应用内引导提示+邮件推送功能教程 | 产品团队 | 自动触发 |
| NPS 0-6（贬损者） | CSM 24 小时内个人跟进电话 | 客户成功 | <24 小时 |
| NPS 7-8（中立者） | 发送满意度提升方案邮件 | 客户成功 | <48 小时 |
| 健康度从绿色降至黄色 | 安排非计划性客户沟通 | 客户成功 | <1 周 |
| 健康度从黄色降至橙色 | 启动风险客户挽留计划 | CS Manager | <48 小时 |
| 健康度降至红色 | 高管关怀+紧急干预 | VP/Director | <24 小时 |

### 7.2 基于生命周期的计划性干预

| 时间点 | 干预动作 | 目标 |
|--------|---------|------|
| 签约后 0-7 天 | 欢迎序列+快速上手指导 | 确保首次价值实现 |
| 签约后 14 天 | "首个里程碑"检查 | 确认核心功能已使用 |
| 签约后 30 天 | 首月价值回顾邮件 | 强化已获得的价值 |
| 签约后 60 天 | 功能深度使用引导 | 提升功能采纳率 |
| 签约后 90 天 | 季度业务回顾（QBR） | 评估ROI、收集反馈 |
| 续费前 90 天 | 续费预热：年度价值报告 | 用数据证明ROI |
| 续费前 60 天 | 续费意向确认 | 识别风险、解决障碍 |
| 续费前 30 天 | 续费提案发送 | 推进续费流程 |
| 续费当天 | 续费确认+感谢 | 完成续费 |
| 续费后 7 天 | 新年度展望+升级建议 | 推进扩展销售 |

### 7.3 QBR（季度业务回顾）结构

```
一、使用数据回顾（10分钟）
   - 关键使用指标和趋势
   - 团队活跃度变化
   - 与同行对比（脱敏后的基准数据）

二、价值实现评估（15分钟）
   - 签约时的目标 vs 当前达成
   - 量化的ROI（节省时间、降低成本、增加收入）
   - 客户自述的价值感受

三、问题与反馈（10分钟）
   - 当前面临的挑战
   - 功能改进建议
   - 服务改进建议

四、下季度规划（10分钟）
   - 新功能/能力培训计划
   - 扩展使用建议（新部门、新场景）
   - 升级/扩展销售机会

五、行动事项（5分钟）
   - 双方明确的下一步行动
   - 责任人和截止日期
```

## 第8节：关键指标及基准

### 8.1 流失与留存核心指标

| 指标 | 定义 | B2B SaaS 健康基准 | 优秀基准 |
|------|------|-------------------|---------|
| 月客户流失率（Logo Churn） | 当月取消客户数 / 月初客户数 | < 2% | < 1% |
| 月收入流失率（Revenue Churn） | 当月流失 MRR / 月初 MRR | < 2% | < 1% |
| 净收入留存率（NRR） | (月初MRR + 扩展 - 流失 - 降级) / 月初MRR | > 100% | > 120% |
| 年客户留存率 | 1 - 年化客户流失率 | > 80% | > 90% |
| 毛流失率（Gross Churn） | 月流失MRR / 月初MRR（不计扩展） | < 3% | < 1.5% |

### 8.2 挽留流程指标

| 指标 | 定义 | 基准范围 | 优化方向 |
|------|------|---------|---------|
| 取消流程挽留率 | 进入取消流程但最终留下的比例 | **25-35%** | 优化挽留方案匹配度 |
| 挽留方案接受率 | 接受挽留方案的比例（不含暂停） | **15-25%** | 提升方案吸引力 |
| 暂停重激活率 | 暂停用户最终重新激活的比例 | **60-80%** | 暂停期间保持沟通 |
| 催收回收率 | Dunning流程成功回收的支付失败比例 | **50-60%** | 优化重试策略和邮件序列 |
| 赢回率 | 取消后 180 天内重新订阅的比例 | **5-15%** | 优化赢回邮件序列 |
| 离开调查完成率 | 取消流程中完成离开调查的比例 | **70-85%** | 简化调查设计 |

### 8.3 客户健康度指标

| 指标 | 定义 | 基准范围 | 说明 |
|------|------|---------|------|
| 健康客户占比（绿色） | 健康度 80-100 分的客户占总客户比例 | > 60% | 可追加销售池 |
| 风险客户占比（橙色+红色） | 健康度 < 60 分的客户占总客户比例 | < 15% | 需立即干预 |
| 风险预测准确率 | 被标记为风险的客户中，实际流失的比例 | > 60% | 模型校准指标 |
| 健康度→流失相关性 | 低健康度客户的实际流失率 vs 高健康度 | 3-5 倍差异 | 验证模型有效性 |

### 8.4 按细分市场的流失基准

| 客户细分 | 月流失率基准 | 说明 |
|----------|-------------|------|
| 企业级（Enterprise） | < 0.5% | 高客单价、长合同、强绑定 |
| 中端市场（Mid-Market） | < 1-2% | 中等客单价、年付为主 |
| 小微企业（SMB） | < 3-5% | 低客单价、月付为主、切换成本低 |
| 自助注册（Self-Serve） | < 5-7% | 最低客单价、纯自助、最高流失 |
| 免费转付费用户 | < 7-10% | 初始留存最差，3 个月后趋于稳定 |

### 8.5 中国 B2B SaaS 流失率参考

| 产品类型 | 参考月流失率 | 特殊因素 |
|----------|-------------|---------|
| CRM/SCRM | 2-4% | 竞争激烈、切换成本相对低 |
| ERP/财务 | 0.5-1.5% | 切换成本极高、数据迁移难 |
| 协作工具 | 3-5% | 免费替代品多（钉钉/飞书） |
| 垂直行业 SaaS | 1-3% | 行业深度绑定、替代品少 |
| 营销 SaaS | 3-5% | 效果波动大、预算敏感 |
| 开发者工具 | 2-4% | 技术替代方案多、开源竞争 |
