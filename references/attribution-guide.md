# 归因分析方法指南 Attribution Guide

> Sloth-MGO-Eido 参考知识库 | 版本 1.0
> 适用范围：B2B SaaS 市场增长运营渠道归因与效果评估

---

## 1. 归因模型总览

### 1.1 三种归因模型对比

| 属性 | 首次触达归因 First-Touch | 末次触达归因 Last-Touch | 多触点加权归因 Multi-Touch Weighted |
|------|------------------------|------------------------|-----------------------------------|
| 核心逻辑 | 100%功劳归于第一个触点 | 100%功劳归于最后一个触点 | 按规则将功劳分配到多个触点 |
| 回答的问题 | 什么渠道带来了新线索？ | 什么渠道促成了转化？ | 哪些渠道组合推动了全旅程？ |
| 适用场景 | 评估获客渠道效率 | 评估转化推动渠道 | 全面评估营销组合 |
| 优点 | 简单清晰，激励拉新 | 简单清晰，激励转化 | 全面公平，接近真实 |
| 缺点 | 忽略培育和转化触点 | 忽略获客和培育触点 | 规则复杂，需充足数据 |
| 推荐使用阶段 | 早期/获客为主阶段 | 短销售周期场景 | 成熟团队/长销售周期 |
| MGO默认优先级 | 辅助参考 | 辅助参考 | **主归因模型** |

## 2. 首次触达归因 First-Touch Attribution

### 2.1 定义与规则

```
归因规则：
  将100%的成交/转化功劳归于线索的第一个已知触点。

首次触达识别逻辑：
  1. 查找该联系人/企业在系统中最早的记录来源
  2. 来源字段 = UTM Source / UTM Medium / UTM Campaign
  3. 如无UTM参数，使用Referrer Header解析
  4. 如无Referrer，标记为"Direct/Unknown"

时间窗口：
  无限制（追溯至最早记录）
```

### 2.2 来源分类标准

| 来源分类 | 识别规则 | 示例 |
|----------|---------|------|
| Organic Search | utm_medium=organic 或 Referrer含搜索引擎域名 | 百度自然搜索、Google Organic |
| Paid Search (SEM) | utm_medium=cpc/ppc | 百度竞价、Google Ads |
| Social Organic | utm_medium=social 且 无付费标记 | 公众号文章、知乎回答、LinkedIn帖子 |
| Social Paid | utm_medium=social-paid 或含广告标记 | 朋友圈广告、信息流广告 |
| Content/SEO | utm_medium=content 或 Referrer=博客/资源页 | 白皮书下载页、博客文章 |
| Email | utm_medium=email | EDM序列、Newsletter |
| Referral | utm_medium=referral 或 Referrer为合作方域名 | 合作伙伴推荐、客户推荐 |
| Event | utm_medium=event | 线下活动、Webinar、展会 |
| Direct | 无Referrer且无UTM | 直接输入URL、书签 |
| Unknown | 数据缺失 | 系统无法识别来源 |

### 2.3 适用报告

- **渠道获客效率报告**：各渠道带来的新线索数量与质量
- **CAC（获客成本）计算**：渠道投入 / 该渠道首触线索数
- **渠道趋势报告**：月度各渠道首触线索数变化趋势

## 3. 末次触达归因 Last-Touch Attribution

### 3.1 定义与规则

```
归因规则：
  将100%的成交/转化功劳归于转化事件发生前的最后一个触点。

末次触达识别逻辑：
  1. 确定转化事件（MQL标记/SQL标记/成交/表单提交等）
  2. 回溯该转化事件前的最近一次营销触点
  3. 该触点 = 末次归因渠道

转化事件定义：
  - Level 1：表单提交（下载/注册/咨询）
  - Level 2：MQL标记（评分达标）
  - Level 3：SQL标记（Sales确认合格）
  - Level 4：Opportunity创建
  - Level 5：Closed-Won成交

回溯时间窗口：
  默认30天。超过30天无触点，标记为"No Recent Touch"。
```

### 3.2 末次触点排除规则

以下触点不计入末次归因（因为不反映营销驱动力）：

| 排除触点 | 理由 |
|----------|------|
| 直接访问（Direct） | 用户记住URL并非营销功劳 |
| 品牌词搜索 | 已有品牌认知，不反映获客渠道 |
| 登录页面访问 | 已是用户行为，非营销触点 |
| 内部IP访问 | 公司内部测试流量 |

排除后的回溯逻辑：跳过排除项，继续向前查找最近的有效营销触点。

### 3.3 适用报告

- **转化驱动力报告**：哪些渠道/活动直接推动了转化发生
- **Campaign ROI报告**：特定Campaign的末次归因成交额/投入
- **着陆页效果报告**：各着陆页的末次归因转化数

## 4. 多触点加权归因 Multi-Touch Weighted Attribution

### 4.1 定义与规则

MGO默认主归因模型。将成交/转化功劳按照触点位置和类型，分配给客户旅程中的多个触点。

### 4.2 权重分配规则

#### 4.2.1 位置权重（Position-Based Weights）

```
基础位置权重模型（U形/浴缸模型）：

  首次触达（First Touch）：30%
  中间触点（Middle Touches）：均分剩余 20%
  线索创建触点（Lead Creation Touch）：20%
  末次触达（Last Touch）：30%

示例：
  客户旅程有5个触点：A → B → C(创建线索) → D → E(转化)
  
  A（首触）：30%
  B（中间）：20% ÷ 2 = 10%
  C（线索创建）：20%
  D（中间）：20% ÷ 2 = 10%
  E（末次）：30%
  
  注意：如果只有1个中间触点，该触点获得全部20%。
  如果没有中间触点（仅2个触点），首末各50%。
  如果首触=线索创建触点，该触点获得 30%+20%=50%。
```

#### 4.2.2 触点类型加权调节（Type-Based Modifier）

在位置权重基础上，根据触点类型施加调节系数：

| 触点类型 | 调节系数 | 说明 |
|----------|---------|------|
| 产品演示请求 | 1.3x | 高意图行为，权重上调 |
| 报价/咨询请求 | 1.3x | 高意图行为 |
| 深度Workshop | 1.2x | 高参与行为 |
| 白皮书下载 | 1.0x | 标准 |
| Webinar参加 | 1.1x | 中高参与 |
| 博客阅读 | 0.8x | 低参与行为，权重下调 |
| 邮件点击 | 0.9x | 中低参与 |
| 社交互动 | 0.7x | 低参与行为 |
| 广告点击 | 0.9x | 付费流量 |

#### 4.2.3 最终权重计算

```
最终权重计算：

  raw_weight(i) = position_weight(i) × type_modifier(i)
  final_weight(i) = raw_weight(i) / Σ raw_weight(j)  （归一化，使总和=100%）

计算示例：
  触点A（首触，博客）：30% × 0.8 = 24%
  触点B（中间，白皮书）：10% × 1.0 = 10%
  触点C（线索创建，Webinar）：20% × 1.1 = 22%
  触点D（中间，邮件）：10% × 0.9 = 9%
  触点E（末次，演示请求）：30% × 1.3 = 39%
  
  总和 = 24+10+22+9+39 = 104
  归一化：A=23.1%, B=9.6%, C=21.2%, D=8.7%, E=37.5%
  
  若该商机成交额=100万：
  A渠道归因=23.1万, B渠道=9.6万, C渠道=21.2万, D渠道=8.7万, E渠道=37.5万
```

### 4.3 时间衰减叠加（可选）

```
在位置权重+类型调节基础上，叠加时间衰减：

  time_factor(i) = e^(-0.02 × days_before_conversion(i))

  days_before_conversion = 该触点距转化事件的天数

此衰减让更接近转化的触点获得额外加成。

完整公式：
  raw_weight(i) = position_weight(i) × type_modifier(i) × time_factor(i)
  final_weight(i) = raw_weight(i) / Σ raw_weight(j)
```

## 5. 归因数据采集要求

### 5.1 数据采集矩阵

| 数据类型 | 采集方式 | 必要字段 | 精度要求 |
|----------|---------|---------|---------|
| 网站行为 | UTM参数 + JS埋点 | source, medium, campaign, content, term, timestamp, session_id, user_id | 页面级+秒级时间戳 |
| 邮件互动 | ESP Webhook回调 | email_id, campaign_id, event_type, timestamp, recipient_id | 事件级 |
| 广告点击 | 广告平台API + 落地页UTM | click_id, campaign_id, ad_group, keyword, cost, timestamp | 点击级 |
| 活动参与 | 活动平台API/手动录入 | event_id, attendee_id, event_type, registration_time, attendance_status | 事件级 |
| 社交互动 | 社交平台API | post_id, interaction_type, user_id, timestamp | 事件级 |
| 电话/会议 | CRM录入 | contact_id, call_type, duration, timestamp, outcome | 通话级 |
| 内容下载 | 表单+CDN日志 | asset_id, asset_type, download_time, form_fields | 下载级 |

### 5.2 UTM规范

```
UTM参数命名规范：

utm_source = 流量来源平台
  命名规则：小写、连字符分隔
  示例：baidu, google, wechat-mp, zhihu, linkedin, email-esp

utm_medium = 流量媒介类型
  标准值：organic, cpc, social, social-paid, email, referral, event, content, display

utm_campaign = 营销活动标识
  命名规则：{年份}{季度}-{活动类型}-{活动名}
  示例：2024q4-webinar-ai-marketing, 2024q3-whitepaper-b2b-trends

utm_content = 内容变体标识（用于A/B测试）
  示例：cta-red, cta-blue, headline-v1, headline-v2

utm_term = 搜索关键词（SEM专用）
  示例：crm系统, 营销自动化工具
```

### 5.3 身份打通（Identity Resolution）

```
线索身份打通优先级：

1. Email地址（主键） — 表单提交时获取
2. 手机号 — 表单/注册时获取
3. Cookie ID → Email映射 — 表单提交时绑定
4. 企业域名聚合 — 同企业邮箱后缀的用户合并
5. IP→企业映射 — 企业IP库反查（精度低，仅辅助）

打通规则：
- 同一Email下的所有Cookie ID归为一个Contact
- 同一企业域名下的多个Contact归为一个Account
- 打通后的归因以Account级别计算（B2B场景）
```

## 6. 常见陷阱与规避

### 6.1 归因分析六大陷阱

| 陷阱 | 描述 | 危害 | 规避方法 |
|------|------|------|---------|
| **过度归因** | 将自然增长归因于营销活动 | 高估营销ROI | 设置对照组；关注增量归因（Incremental Attribution） |
| **渠道蚕食** | 品牌词SEM吃掉自然搜索流量 | 为已有流量多付广告费 | 定期做品牌词SEM开/关测试 |
| **Cookie断裂** | 跨设备/跨浏览器导致旅程断裂 | 低估多触点渠道价值 | 尽早引导登录/邮箱绑定 |
| **幸存者偏差** | 只分析成交客户的旅程 | 忽略流失路径中的问题渠道 | 同时分析Lost Deal的归因路径 |
| **时间窗口偏差** | 归因窗口太短或太长 | 漏掉早期触点或引入噪声 | 根据平均销售周期设置窗口（建议1.5x平均周期） |
| **内部流量污染** | 内部员工行为被计入归因 | 数据失真 | 排除内部IP + 内部邮箱域名 |

### 6.2 数据质量检查清单

```
月度数据质量检查（Data Quality Audit）：

□ UTM覆盖率：有UTM标记的线索占比是否>80%
□ 来源Unknown率：Unknown/Direct来源占比是否<15%
□ 触点完整度：平均每个成交客户可追踪触点数是否>3个
□ 时间戳准确性：抽样检查时间戳是否合理（无未来时间/异常早期时间）
□ 身份打通率：Cookie ID成功映射到Email的比率是否>60%
□ 重复数据：去重后的触点记录与原始记录差异是否<5%
□ 内部流量排除：内部IP/域名是否已正确排除
```

## 8. 数据缺失降级处理

当归因数据不完整时，按以下规则降级处理：

### 8.1 降级优先级

```
归因模型降级链：

Level 1（最优）：多触点加权归因
  要求：完整的多触点旅程数据（≥3个触点/客户）
  ↓ 数据不足时降级
Level 2：首触+末触双归因
  要求：至少有首触和末触数据
  ↓ 数据不足时降级
Level 3：末次触达单归因
  要求：至少有转化前一个触点数据
  ↓ 数据不足时降级
Level 4：来源字段归因
  要求：线索记录中有来源字段（Source Field）
  ↓ 数据不足时降级
Level 5：标记为"Unattributed"
  无可用归因数据，计入"未归因"类别
```

### 8.2 各场景降级规则

| 缺失场景 | 影响 | 降级处理 | 补救措施 |
|----------|------|---------|---------|
| 无UTM参数 | 无法识别具体来源 | 使用Referrer Header推断 → 推断失败标记Direct | 强制所有投放链接带UTM |
| Cookie被清除 | 旅程断裂 | 以表单提交作为新旅程起点 | 引导登录绑定Email |
| 跨设备访问 | 同一用户识别为多人 | Email匹配合并 → 无Email则独立计算 | 鼓励注册/登录 |
| 线下活动无追踪 | 活动触点丢失 | SDR手动录入标记来源=Event | 活动签到使用带UTM的二维码 |
| CRM录入不规范 | Sales触点缺失 | 仅使用自动采集数据 | CRM必填字段+录入培训 |
| 历史数据缺失 | 早期触点不可追溯 | 以系统内最早记录作为"首触" | 标注"数据起始于[日期]" |

### 8.3 数据质量阈值

```
归因报告可信度分级：

高可信度（Green）：
  - UTM覆盖率 ≥ 80%
  - Unknown来源 ≤ 10%
  - 平均可追踪触点 ≥ 4个/客户
  - 身份打通率 ≥ 70%

中可信度（Yellow）：
  - UTM覆盖率 60%-79%
  - Unknown来源 11%-20%
  - 平均可追踪触点 2-3个/客户
  - 身份打通率 50%-69%

低可信度（Red）：
  - UTM覆盖率 < 60%
  - Unknown来源 > 20%
  - 平均可追踪触点 < 2个/客户
  - 身份打通率 < 50%
  - → 报告需标注"数据质量不足，结论仅供参考"
```
