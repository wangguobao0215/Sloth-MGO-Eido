## 1. 广告法违禁词清单 Prohibited Terms

依据《中华人民共和国广告法》第九条、第二十八条及市场监管总局相关解释，以下词汇在营销内容中严格禁止使用或需替换处理。

### 1.1 绝对化用语（Absolute Claims）

| 类别 | 违禁词列表 |
|------|-----------|
| 最高级表述 | 最好、最佳、最优、最优秀、最先进、最高级、最高端、最高档、最低价、最便宜、最大、最小、最新、最快、最强、最具、最赚、最时尚 |
| 第一类表述 | 第一、第一品牌、全国第一、行业第一、排名第一、销量第一、No.1、TOP1、首选、首家、首创、独家 |
| 唯一类表述 | 唯一、独一无二、绝无仅有、前所未有、史无前例、万无一失、无与伦比、无可比拟、无可替代 |
| 极限类表述 | 极致、极品、极佳、终极、顶级、顶尖、尖端、冠军、王牌、王者、领导者、缔造者、开创者、领航者、领袖 |
| 百分比绝对 | 100%、100%好评、100%满意、100%安全、100%有效、零风险、零缺陷、零投诉、零差评、零故障 |
| 国家级/世界级 | 国家级、世界级、全球级、宇宙级、国家领导人推荐、国家XX机关推荐、特供、专供、指定 |
| 权威性暗示 | 驰名商标、老字号（未经认定）、金牌、银牌、名牌、优等品、免检、质量免检、无需国家质量检测 |
| 时间极限 | 永久、永恒、万能、祖传、特效、无敌、纯天然、仅此一次、错过再无 |

### 1.2 虚假承诺用语

| 类别 | 违禁词列表 |
|------|-----------|
| 效果承诺 | 保证见效、药到病除、一次搞定、立竿见影、包治百病、根治、彻底解决、永不复发 |
| 收益承诺 | 稳赚不赔、躺着赚钱、日赚XX万、保本保息、翻倍增长（无数据支撑）、高额回报 |
| 时间承诺 | 3天见效、7天上手、一周回本、一个月翻倍（无合理依据） |
| 对比诋毁 | 秒杀XX（竞品名）、XX不行我们行、吊打XX、碾压竞品、完胜同行 |

### 1.3 B2B SaaS 行业高频风险词

| 风险词 | 风险等级 | 合规替换建议 |
|--------|---------|-------------|
| 行业领先 | 黄色 | "行业前列" 或提供第三方排名来源 |
| 市场占有率第一 | 红色 | 需引用权威第三方报告+时间范围+细分市场 |
| 客户满意度100% | 红色 | "客户满意度达XX%（来源：XX调研，样本N=XX）" |
| 零宕机 | 红色 | "99.99% SLA可用性保障" |
| 安全无忧 | 黄色 | "通过XX安全认证，符合XX标准" |
| 降本增效 | 黄色 | 需附具体数据："平均降低XX%运营成本（基于N家客户数据）" |
| 开箱即用 | 黄色 | "标准部署周期X天，提供实施指导" |
| 一站式解决方案 | 黄色 | 明确覆盖范围，避免暗示全能 |
| 全行业通用 | 红色 | 列出已验证行业清单 |
| 智能/AI驱动 | 黄色 | 需说明具体AI技术及应用场景 |

## 2. 虚假宣传识别模式 False Advertising Patterns

### 2.1 六大虚假宣传模式

| 模式 | 定义 | 典型案例 | 检测规则 |
|------|------|---------|---------|
| 数据造假 | 使用无来源/过时/篡改数据 | "帮助10000+企业实现增长"（实际客户200家） | 要求所有数据标注来源、时间、统计口径 |
| 资质冒用 | 声称未获得的认证/奖项 | "ISO27001认证"（仅在申请中） | 核验证书编号及有效期 |
| 案例虚构 | 编造或夸大客户案例 | 未经客户授权使用Logo及数据 | 要求客户授权书或公开信息佐证 |
| 功能夸大 | 承诺尚未实现的功能 | Roadmap功能描述为已上线 | 区分"已上线"vs"规划中"标注 |
| 对比误导 | 选择性对比/不公平对比 | 用对手旧版本vs自己新版本对比 | 对比需注明版本、时间、条件 |
| 暗示背书 | 暗示与知名品牌/机构合作 | "XX大厂同款技术"（无合作关系） | 合作声明需合同或公开声明佐证 |

### 2.2 检测触发规则

```
IF 内容包含 数字型声明 (如 "XX%增长", "XX家客户"):
  → 触发数据源核验：要求提供来源 + 统计时间 + 统计口径
IF 内容包含 品牌名称 (非自身):
  → 触发竞品提及审核（见第6节）
IF 内容包含 认证/奖项声明:
  → 触发资质核验：要求证书编号或官方链接
IF 内容包含 客户Logo/名称:
  → 触发授权核验：要求授权证明文件编号
```

## 3. PIPL 个人信息保护规则

依据《个人信息保护法》（2021年11月1日施行），B2B 营销场景下的个人信息处理规范：

### 3.1 信息采集规则

| 场景 | 可采集字段 | 采集条件 | 禁止行为 |
|------|-----------|---------|---------|
| 官网表单 | 姓名、企业名、职位、企业邮箱、手机号 | 明示告知 + 勾选同意 + 隐私政策链接 | 预勾选同意框、捆绑授权 |
| 活动报名 | 姓名、企业、职位、联系方式、参会需求 | 单独同意 + 明示用途 | 超范围采集（如身份证号） |
| 内容下载 | 企业邮箱、姓名、企业名 | 告知用于发送内容 + 后续营销需单独授权 | 一次授权覆盖全部后续营销 |
| 线上直播 | 姓名、企业、职位 | 注册时告知 + 同意条款 | 未告知录制或数据共享 |
| Cookie/埋点 | 设备ID、浏览行为、IP地址 | Cookie Banner + 分类同意 | 未提供拒绝选项 |

### 3.2 信息存储规则

| 要求 | 标准 |
|------|------|
| 存储位置 | 中国境内服务器（境外传输需安全评估） |
| 存储期限 | 实现处理目的所需最短时间，超期自动删除或匿名化 |
| 加密标准 | 传输层 TLS 1.2+，存储层 AES-256 加密 |
| 访问控制 | 最小权限原则，操作日志留存6个月+ |
| 备份策略 | 加密备份，与生产环境同等保护级别 |

### 3.3 信息使用规则

| 场景 | 合规做法 | 违规做法 |
|------|---------|---------|
| EDM 营销 | 已获邮件营销单独授权 + 提供退订链接 | 购买邮件列表群发 |
| 电话外呼 | 已获电话联络授权 + 工作时间呼叫 | 无授权Cold Call、非工作时间骚扰 |
| 数据共享 | 告知接收方 + 单独同意 + 数据处理协议 | 未经同意转给第三方 |
| 用户画像 | 告知画像逻辑 + 提供退出机制 | 自动化决策未告知且无人工介入选项 |
| 数据删除 | 收到请求15个工作日内响应 | 不提供删除通道或拖延处理 |

### 3.4 数据脱敏规则

| 字段类型 | 脱敏方式 | 示例 |
|----------|---------|------|
| 手机号 | 中间4位掩码 | 138****5678 |
| 邮箱 | @前保留首尾字符 | w***o@company.com |
| 姓名 | 保留姓，名掩码 | 王** |
| 企业名 | 分析场景全脱敏，展示场景保留（需授权） | 某制造业客户A |
| IP地址 | 末段掩码 | 192.168.1.*** |
| 身份证号 | 仅保留前3后4位 | 310***********1234 |

## 4. 数据安全法要点

依据《数据安全法》（2021年9月1日施行），MGO 运营相关核心义务：

### 4.1 数据分类分级

| 级别 | 定义 | MGO 场景示例 | 保护要求 |
|------|------|-------------|---------|
| 核心数据 | 关系国家安全的数据 | 通常不涉及 | 国家审批 |
| 重要数据 | 影响公共利益的数据 | 大规模用户行为聚合数据（100万+） | 安全评估 + 年度风险评估 |
| 一般数据 | 企业经营数据 | CRM线索数据、营销分析数据、内容数据 | 基本安全措施 |

### 4.2 营销数据处理红线

- **禁止**：未经授权爬取竞品网站用户数据
- **禁止**：将客户数据用于未告知目的（如转售给第三方广告平台）
- **禁止**：在营销物料中暴露客户商业敏感数据（如营收、成本）
- **要求**：数据处理活动记录留存3年以上
- **要求**：重要数据处理者需指定数据安全负责人
- **要求**：发生数据泄露需72小时内报告主管部门

## 5. 行业特定合规规则

### 5.1 金融行业（FinTech SaaS）

| 规则 | 详细要求 |
|------|---------|
| 资质声明 | 必须注明金融业务许可证编号 |
| 收益表述 | 禁止承诺收益，必须标注"投资有风险" |
| 客户案例 | 金融机构案例需获得书面授权，不得暗示监管背书 |
| 数据安全 | 符合银保监会数据安全管理办法，等保三级+ |
| 算法披露 | 涉及风控/信贷模型需说明算法逻辑 |

### 5.2 医疗健康行业（HealthTech SaaS）

| 规则 | 详细要求 |
|------|---------|
| 效果声明 | 严禁治愈性承诺，软件不得声称为医疗器械（除非取得注册证） |
| 患者数据 | 符合《健康医疗大数据标准》，不得在案例中暴露患者信息 |
| 资质引用 | 需注明医疗器械注册证号/备案号（如适用） |
| 广告审查 | 医疗相关内容需提交卫健委广告审查 |
| 禁用词 | "治愈率XX%"、"替代医生诊断"、"精准医疗"（无依据） |

### 5.3 教育行业（EdTech SaaS）

| 规则 | 详细要求 |
|------|---------|
| 效果承诺 | 禁止"保过"、"保分"、"提分XX分"等承诺 |
| 师资声明 | 教师资质需真实可查，禁止虚构名师 |
| 未成年保护 | 面向未成年人需获得监护人同意，限制数据采集范围 |
| 价格透明 | 禁止"原价XX现价XX"无依据标注，禁止虚构倒计时 |
| 退费政策 | 必须明示退费条款，符合预付费管理规定 |

## 6. 竞品提及红线（反不正当竞争法）

依据《反不正当竞争法》第十一条，竞品提及规范：

### 6.1 三级行为界定

| 级别 | 行为 | 判定 | 示例 |
|------|------|------|------|
| 红色阻断 | 捏造/散布虚假信息贬损竞品 | 违法 | "XX产品频繁宕机导致客户损失"（无依据） |
| 红色阻断 | 使用竞品注册商标进行SEM投放 | 违法 | 购买竞品品牌词做广告 |
| 红色阻断 | 虚假对比：使用过时/错误竞品数据 | 违法 | 用对手2年前版本功能做对比表 |
| 黄色警告 | 暗示性贬损（"不像某些产品…"） | 高风险 | "不像市面上其他产品，我们真正做到了…" |
| 黄色警告 | 未注明来源的竞品数据引用 | 高风险 | 引用竞品营收数据无出处 |
| 绿色通过 | 基于公开信息的客观功能对比 | 合规 | 对比表注明"数据来源：各厂商官网，截至2024年Q3" |
| 绿色通过 | 引用第三方报告中的竞品信息 | 合规 | "据Gartner 2024年报告…" |

### 6.2 竞品内容审核流程

```
1. 检测：内容中是否出现竞品品牌名/产品名
2. 分类：直接点名 vs 暗示指代 vs 品类泛指
3. 语境分析：陈述事实 vs 价值判断 vs 贬损攻击
4. 来源核验：所有竞品相关声明是否有公开来源
5. 结论输出：绿色通过 / 黄色修改建议 / 红色阻断
```

## 7. 合规三级结果定义

所有内容在发布前经过合规引擎审核，输出以下三级结果：

### 7.1 红色阻断 RED-BLOCK

| 属性 | 说明 |
|------|------|
| 含义 | 内容包含法律违规风险，**立即阻断发布** |
| 触发条件 | 广告法违禁词（绝对化用语）、虚假宣传、竞品诋毁、无授权使用客户信息、违反PIPL核心条款 |
| 处理流程 | 阻断发布 → 标记违规点 → 提供修改建议 → 修改后重新审核 |
| 升级规则 | 同一内容3次红色阻断 → 升级至法务部人工审核 |
| 记录要求 | 记录违规内容快照、审核时间、审核结果、修改历史 |

### 7.2 黄色警告 YELLOW-WARN

| 属性 | 说明 |
|------|------|
| 含义 | 内容存在合规风险但不构成明确违法，**建议修改后发布** |
| 触发条件 | 行业敏感词、数据无来源标注、夸大性表述（如"行业领先"无佐证）、竞品暗示性对比 |
| 处理流程 | 标记风险点 + 修改建议 → 作者确认/修改 → 可选择带风险发布（需负责人审批） |
| 升级规则 | 带风险发布需营销总监审批签字 |
| 记录要求 | 记录风险点、修改建议、作者处理决策 |

### 7.3 绿色通过 GREEN-PASS

| 属性 | 说明 |
|------|------|
| 含义 | 内容无合规风险，**可直接发布** |
| 通过条件 | 无违禁词、数据有来源、竞品提及合规、个人信息处理合规、行业规则符合 |
| 处理流程 | 自动通过 → 记录审核日志 → 发布 |
| 有效期 | 审核结果有效期30天，超期需重新审核 |
| 记录要求 | 记录审核时间、内容版本号、审核引擎版本 |

## 8. 合规审核执行流程

```
内容提交
  ↓
[Layer 1] 违禁词扫描（自动）
  → 命中绝对化用语 → RED-BLOCK
  → 命中敏感词 → YELLOW-WARN
  → 通过 → 进入 Layer 2
  ↓
[Layer 2] 语义合规分析（AI辅助）
  → 虚假宣传模式检测
  → 竞品提及合规检测
  → 效果承诺检测
  → 输出风险等级
  ↓
[Layer 3] 行业特定规则（规则引擎）
  → 金融/医疗/教育行业叠加规则
  → PIPL个人信息规则
  → 输出最终结果
  ↓
RED-BLOCK → 阻断 + 修改建议
YELLOW-WARN → 警告 + 修改建议 + 可带风险发布
GREEN-PASS → 通过 + 审核日志
```

## 10. 外部AI服务调用脱敏规则

在调用外部AI服务时，输入端必须执行自动脱敏，防止敏感业务信息泄露至第三方。

### 10.1 适用场景

凡涉及将内部数据作为输入发送至外部AI服务的场景，均须执行本规则：

- GPT / Claude 等大语言模型 API
- 翻译 API（DeepL、Google Translate 等）
- 图片生成 / 多模态 API
- 语音转文字 / 文字转语音 API
- 其他第三方 AI SaaS 服务

### 10.2 脱敏字段清单

| 字段类型 | 脱敏方式 | 示例 |
|----------|---------|------|
| 客户公司名称 | 替换为顺序占位符 | 某科技有限公司 → `[Company_A]`，另一家 → `[Company_B]` |
| 客户联系人姓名 | 替换为顺序占位符 | 王明 → `[Contact_1]`，李华 → `[Contact_2]` |
| 金额 / 合同价值 | 替换为占位符 | 500万 → `[Amount_X]` |
| 内部产品代号 / 未发布功能名 | 替换为占位符 | Project Phoenix → `[Product_X]` |
| 手机号 / 邮箱 / 微信号 | 完全移除或替换为占位符 | 13800001234 → `[PHONE_REMOVED]`，wx_id → `[WECHAT_REMOVED]` |
| 内部组织架构信息 | 替换为占位符 | 华东大区三部 → `[Dept_X]` |

### 10.3 脱敏时机

在构造外部 API 请求 **之前** 自动执行脱敏处理。流程如下：

```
原始输入
  ↓
[脱敏引擎] 扫描并替换敏感字段，生成「占位符 ↔ 真实值」映射表
  ↓
脱敏后的输入 → 发送至外部 AI 服务
```

映射表仅保存在本地内存中，不随请求发送，不持久化至外部存储。

### 10.4 还原机制

外部 AI 服务返回结果后，自动执行反向还原：

```
AI 返回结果（含占位符）
  ↓
[还原引擎] 根据映射表将占位符替换回真实值
  ↓
最终输出呈现给用户
```

还原失败（如占位符被 AI 改写或丢失）时，标注 `[还原失败，请人工核对]` 并保留占位符原文。

### 10.5 免脱敏白名单

以下类型的信息可不执行脱敏，但须在 `config.yaml` 的 `desensitization_whitelist` 字段中显式配置：

- 上市公司名称（公开信息）
- 已正式发布的产品名称
- 公开可查的行业术语、标准名称
- 用户手动标注为"可公开"的信息

未在白名单中的信息一律执行脱敏，宁可过度脱敏也不遗漏。
